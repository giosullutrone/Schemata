# Schemata Audit — 2026-03-05

## Bugs / Issues to Fix

### 1. `localStorage` access unprotected
**Files:** `src/App.tsx`, `src/store/useCanvasStore.ts`

Reading color mode and sidebar state from `localStorage` can throw in private browsing mode or when storage is full. Wrap all `localStorage.getItem`/`setItem` calls in try/catch with sensible fallbacks.

### 2. `saveAllFiles()` race condition
**File:** `src/store/useCanvasStore.ts`

Reads the `files` snapshot at the start of the function, then sequentially awaits writes to disk. If the user edits a file while the save is in progress, the stale version overwrites the newer in-memory edits.

**Fix:** Snapshot each file's content immediately before writing it, or lock edits during save.

### 3. `moveFileToFolder` / `createFileInFolder` TOCTOU race
**Files:** `src/store/useCanvasStore.ts`, `src/utils/fileIO.ts`

Both check if a file exists in the target folder, then create it. Another process (or browser tab) could create the file between the check and the create, causing a silent overwrite.

**Mitigation:** Low risk in single-user browser context, but worth noting. The File System Access API doesn't offer atomic create-if-not-exists.

### 4. Viewport floating-point drift
**File:** `src/store/useCanvasStore.ts` (`saveViewport`)

Uses strict `===` for zoom comparison. Floating-point imprecision from repeated zoom operations could cause unnecessary file updates.

**Fix:** Use epsilon comparison (e.g., `Math.abs(a - b) < 0.0001`).

### 5. No undo for drag operations
**File:** `src/store/useCanvasStore.ts` (`setCanvasNodes` / `setCanvasEdges`)

These actions are called by ReactFlow during drag/select operations. They don't call `pushUndo()`, so dragging a node can't be undone with Ctrl+Z. This is the most noticeable undo/redo gap.

### 6. No file handle invalidation recovery
**File:** `src/store/useCanvasStore.ts`

If the File System Access API permission is revoked (e.g., tab goes stale, browser restarts), file handles become invalid. There's no mechanism to re-acquire them short of re-opening the folder.

**Fix:** Catch permission errors on save and prompt the user to re-open the folder.

### 7. `moveFileToFolder` clears undo/redo stacks
**File:** `src/store/useCanvasStore.ts`

Moving a file wipes all undo history with no warning. Users don't expect this behavior.

## Architectural Observations

- **Undo/redo is snapshot-based** using `JSON.parse(JSON.stringify(...))` — O(n) per action. Fine at current scale, could become a bottleneck with very large diagrams.
- **Error state is a single string** — Multiple errors overwrite each other. A queue or toast system would be more robust.
- **Dirty tracking uses `JSON.stringify` comparison** — Expensive on large files during undo/redo. Could use a generation counter instead.
- **ID counters are module-level globals** — Safe for single-instance usage but not idempotent if multiple instances share the same module scope.
- **No conflict resolution for multi-file undo/redo** — Undo stack is per-file-path, but the UI doesn't indicate which file was affected by an undo operation.
